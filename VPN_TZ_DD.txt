1. ОБЩЕЕ ОПИСАНИЕ ПРОЕКТА

Цель проекта:
Создать Telegram SaaS-сервис для продажи VPN-подписок через Telegram-бота с автоматической оплатой, автоматической выдачей VPN-ключей, хранением пользователей, тарифов, платежей, подписок и наличием административной панели.

Формат работы системы:
- пользователь взаимодействует только через Telegram-бота
- бот не работает с БД напрямую
- бот обращается к Go backend по HTTP API
- Go backend обрабатывает бизнес-логику
- Go backend работает с БД, оплатами, VPN provider и админкой

2. ЦЕЛИ ПРОЕКТА

2.1 Основные цели
- продавать VPN-подписки через Telegram
- выдавать VPN-ключ автоматически после оплаты
- управлять тарифами
- управлять пользователями и подписками
- иметь backend, пригодный для production
- иметь архитектуру, которую можно масштабировать

2.2 Дополнительные цели
- обеспечить удобный деплой на VPS

7.5 Получение VPN ключа
Сценарий:
- пользователь нажимает Получить VPN ключ
- бот отправляет запрос на backend
- backend проверяет активную подписку
- backend проверяет, есть ли уже активный ключ
- если ключ есть — отдаёт его
- если ключа нет — создаёт его у VPN provider
- backend сохраняет ключ
- backend возвращает access_url
- бот показывает его пользователю

Результат:
- пользователь получает рабочий VPN ключ

7.6 Истечение подписки
Сценарий:
- фоновая задача backend проверяет просроченные подписки
- backend меняет статус active -> expired
- backend деактивирует ключ у VPN provider
- backend помечает VPNKey как неактивный
- бот может отправить уведомление

Результат:
- просроченные доступы отключаются автоматически

12.1 Цель
Создать реальную оплату тарифа с автоматическим подтверждением через webhook.

12.2 Что должно работать
- create payment
- redirect/confirmation url
- webhook payment.succeeded
- автоматическое создание подписки

12.3 Что хранить в Payment
- внутренний id
- external_payment_id
- payment_url
- status
- raw_response

12.4 Что делает backend при webhook
- получает event
- проверяет payment id
- находит Payment
- если статус ещё не обработан:
  - меняет статус на succeeded
  - создаёт подписку

13.3 Логика выдачи ключа
- если подписки нет — ошибка

- если подписка есть и ключа нет — создать новый
- сохранить в БД
- вернуть пользователю

22. ИТОГ

Проект строится по модели:
Python bot = интерфейс
Go backend = ядро

Главный принцип:
бот ничего не знает о базе и не решает бизнес-задачи.
Всё важное находится в backend.

Это позволяет:
- не запутаться
- легко масштабировать проект
- удобно деплоить
- безопасно подключать оплаты и VPN
